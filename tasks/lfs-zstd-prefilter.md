# Task: Git LFS zstd compression extension

## End Goal

A standalone Go binary (`lfs-zstd-filter`) registered as a **git-lfs
extension** (not a `filter.process`). It acts as a transparent
stdin→stdout transformer:

- **clean** (`lfs-zstd-filter clean %f`): reads raw file bytes from stdin,
  compresses them with zstd, writes the compressed bytes to stdout. LFS stores
  those bytes locally and uploads them to the server.
- **smudge** (`lfs-zstd-filter smudge %f`): reads compressed bytes from stdin
  (streamed out of the LFS local object store), decompresses them, writes the
  original bytes to stdout. Git writes those to the working tree.

LFS handles all pointer management (OID computation, `ext-*` pointer lines,
local object storage, server upload/download). Stock LFS servers are unaware
of compression; they store and serve ordinary LFS objects whose content happens
to be a zstd-compressed stream.

### Extension protocol (from `git-lfs/lfs/extension.go` + `git-lfs/docs/extensions.md`)

LFS invokes the extension as a subprocess with the filename as an argument:
```
# clean: raw file bytes → stdin; compressed bytes ← stdout
lfs-zstd-filter clean path/to/file

# smudge: compressed bytes → stdin; original bytes ← stdout
lfs-zstd-filter smudge path/to/file
```
- Must exit 0 on success, non-zero on failure.
- Errors go to stderr; git-lfs surfaces them to the user.
- Must be a **pure stdin→stdout** transform with no side effects.
- On smudge, extensions are invoked in **reverse priority order** to undo
  what clean did.

### Pointer file produced by git-lfs when an extension is active

```
version https://git-lfs.github.com/spec/v1
ext-0-zstd sha256:<sha256-of-original-uncompressed-file>
oid sha256:<sha256-of-compressed-file>
size <compressed-size-in-bytes>
```

The `ext-0-zstd` line is written by git-lfs (not by this binary) only when
the compressed OID differs from the original OID. git-lfs uses it on smudge
to verify that the extension correctly reconstructed the original content.

### Required git config

It is highly recommended to install the extension locally (per-repository):
```sh
lfs-zstd-filter install
```

This writes to `.git/config`:
```ini
[lfs "extension.zstd"]
  clean    = lfs-zstd-filter clean %f
  smudge   = lfs-zstd-filter smudge %f
  priority = 0
```

### `.gitattributes` (unchanged from standard LFS usage)

```
*.bin filter=lfs diff=lfs merge=lfs -text
```

`filter=lfs` causes git-lfs to run its clean/smudge pipeline, which in turn
invokes registered extensions including this one.

## Constraints

- **clean is unconditional**: every file, regardless of its existing format
  (including `.zst`, `.gz`, `.jpg`), is wrapped in a zstd frame. This is the
  only correct behaviour — see Failed Attempts below.
- **smudge migration path**: objects committed *before* the extension was
  enabled were stored uncompressed, or might be pre-extension `.zst` files. On smudge those objects must be passed
  through unchanged. A custom zstd skippable frame (`0x50 0x2A 0x4D 0x18 0x04 0x00 0x00 0x00 L F S Z`) determines
  whether the incoming bytes are generated by this filter.
- **Empty files**: size 0 input → size 0 output (zstd handles this naturally).
- **LFS pointer pass-through**: git-lfs never sends pointer text through
  extensions, so this binary does not need to detect or handle pointer files.

## Failed Attempts

### Magic-byte guard in `clean` (idempotency guard)

**What was tried:** `clean` peeked at the first 4 bytes and skipped compression
if the input already started with the zstd frame magic (`0x28 0xB5 0x2F 0xFD`).
The rationale was to avoid double-compressing files that had already been
through the filter.

**Why it was wrong:**

1. **It corrupted real user files.** Any file whose content legitimately begins
   with the zstd magic — `.zst` files, and any other file that starts with
   those bytes — would be stored uncompressed by `clean`, then *decompressed*
   by `smudge`, producing garbage on checkout.

2. **The problem it was solving does not exist.** git-lfs calls `clean` exactly
   once per `git add` on the raw working-tree bytes. There is no code path in
   which a file that has already been through this filter's `clean` arrives at
   `clean` again, so there is nothing to guard against.

**Fix:** removed the magic-byte check from `clean` entirely. `clean` is now a
single unconditional `compress.Compress(dst, src)` call.

## Implementation

### File layout

`cmd/lfs-zstd-filter/main.go`: binary entry point; clean + smudge subcommands
`internal/compress/compress.go`: Compress, Decompress, IsZstd helpers

### `clean` behaviour

Unconditionally compresses stdin to stdout using zstd at `SpeedDefault` level,
prepended by a 12-byte zstd skippable frame (`0x50 0x2A 0x4D 0x18 0x04 0x00 0x00 0x00 L F S Z`).
This ensures `smudge` can uniquely identify files processed by this filter, avoiding
accidental decompression of pre-existing `.zst` files that were pushed to LFS before
the extension was enabled. For already-compressed or otherwise incompressible data
zstd produces a frame that is marginally larger than the input — this is acceptable
and unavoidable. If compression ratio matters for specific file types, exclude them
from LFS tracking via `.gitattributes` rather than inside this binary.

### `smudge` behaviour

Peeks at the first 12 bytes of stdin:
- **LFSZ skippable frame present** → decompress the full stream to stdout. The zstd decoder naturally skips the frame.
- **LFSZ skippable frame absent** → copy bytes through unchanged. This is the migration
  path for LFS objects that were stored before the extension was enabled, including
  raw pre-extension `.zst` files.

### Migration edge case (pre-extension `.zst` files)

Since we now prepend a unique `LFSZ` skippable frame, `smudge` correctly
identifies pre-extension `.zst` files because they start with the standard
zstd magic, not our `LFSZ` skippable magic. They are safely passed through
unchanged. After the next `git add` with the extension active, the file
will be stored as `[LFSZ skippable frame] + zstd(original-.zst-bytes)` and
will round-trip correctly from then on.
